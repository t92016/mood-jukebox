## 下一步（需要使用者手動做的 3 件事）
1. **申請 Groq API Key**：https://console.groq.com/keys （免費，Google 登入即可）
2. **申請 YouTube Data API v3 Key**：Google Cloud Console → 建立專案 → 啟用「YouTube Data API v3」→ 憑證 → API 金鑰
3. **建立 Firebase 專案**：Firebase Console → 新增專案 → 建立 Firestore（生產模式）→ 註冊 Web App → 把設定貼進 `public/js/firebase-config.js`，並把 `firestore.rules` 貼到 Firestore 規則頁發布

完成後告訴 AI，接著做：
- [ ] `npm install`
- [ ] `.env` 填入兩把 Key（本機測試用）
- [ ] `npm run dev` 本機全流程測試（點歌 → 快取 → 播放 → 點唱牆）
- [ ] `npm run seed` 預填 36 首快取
- [ ] 建立 Netlify 站台、設定 3 個環境變數（GROQ_API_KEY / YOUTUBE_API_KEY / 不需 Firebase，因為走前端）、連 GitHub repo 自動部署
- [ ] 手機寬度（≤480px）自我檢查

## 重大技術決策
| 決策 | 理由 |
|---|---|
| 用 Netlify Functions 而非純前端 | 規劃書缺後端；Groq Key 放前端會被秒偷，必須代理 |
| 用 Netlify 而非 GitHub Pages | Pages 純靜態跑不了 Functions；Netlify 免費額度足夠且連 repo 自動部署 |
| GitHub repo 設 private | Netlify 免費版支援 private repo，兒童專案保守為上（develop skill 的 public 慣例是為了 Pages，本案不適用） |
| `Users` 集合改名 `MoodLogs` | 無帳號系統，匿名日誌用 MoodLogs 名實相符 |
| Firestore 前端直連（不經代理） | Firebase client SDK + Security Rules 本來就設計給前端用；多一層代理只增延遲 |
| Firebase 未設定時優雅降級 | `firebase-config.js` 沒填也能點歌（不寫紀錄），方便先測 UI |
| 種子歌放 `seeds/songs-seed.json` 不寫死 videoId | videoId 由 seed 腳本實際呼叫 API 取得，不假造 |
