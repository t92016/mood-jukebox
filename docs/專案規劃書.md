# 專案規劃書：心情點唱機 (Mood Jukebox)

## 1. 專案概述
- **專案目標**：開發一款結合 AI 情緒分析與雲端資料庫的音樂推薦網頁應用。
- **目標受眾**：主要針對國小高年級學生（如蘆洲忠義國小的學童）設計，介面需具備親和力與易用性。
- **核心技術棧**：
  - AI 模型：Groq API (負責極速情緒分析與推薦)
  - 資料庫：Firebase Firestore (負責儲存心情紀錄與歌曲資料快取)
  - 音樂來源：YouTube Data API v3 & IFrame Player API
  - 前端開發：HTML, CSS, JavaScript 

## 2. 介面設計規範 (UI/UX)
- **視覺風格**：頁面設計要以**溫馨、舒服、可愛的日式風格**為主，可融入日系手繪、簡約或卡通化元素，確保國小學生看了會喜歡且願意主動使用。
- **互動設計**：操作流程需直覺，減少複雜的輸入步驟，文字說明需簡單明瞭，減少孩童的認知負擔。

## 3. 系統資料流架構
1. **使用者輸入**：接收使用者的「心情描述」。
2. **AI 分析與推薦**：將心情文字加上系統提示詞送給 Groq API，限制模型僅回傳結構化 JSON 格式的推薦歌單與歌手。
3. **快取檢查 (Firestore)**：在呼叫 YouTube API 前，先至 Firebase `SongsCache` 檢查是否已有該歌曲的 YouTube ID，避免浪費 API 搜尋配額。
4. **搜尋影片 (YouTube API)**：若無快取，則呼叫 YouTube Data API 搜尋「歌手 + 歌名」取得首選的影片 ID (`videoId`)，並將結果存入 Firestore 快取池。
5. **資料寫入**：將使用者的心情日誌、推薦歌曲、YouTube ID 與時間戳記寫入 Firestore 的 `Users` 集合中。
6. **前端播放**：透過 YouTube IFrame API 在網頁上無縫嵌入並播放取得的影片，結合 UI 呈現心情點唱牆。

## 4. 核心挑戰與避坑指南
- **Groq API 輸出控制**：必須在 System Prompt 強制規定只能輸出純 JSON (例如：`{"song": "...", "artist": "...", "reason": "..."}`)，避免後端解析錯誤 (Parse Error)。提示詞內建議限制優先推薦能在 YouTube 上輕易找到的流行歌曲。
- **YouTube 配額陷阱 (嚴重注意)**：YouTube API 搜尋功能極耗配額 (一次搜尋消耗 100 單位)。**務必先實作 Firebase 歌曲快取機制**，這是維持免費營運的絕對關鍵。
- **資料庫結構建議**：
  - `SongsCache` (Collection): 存放 `歌名+歌手` 作為 Document ID 或欄位，對應其 `YouTube ID`。
  - `Users` (Collection): 存放個別使用者的歷次心情紀錄與推薦結果。

## 5. 開發階段建議 (供 AI Agent 執行參考)
- **Phase 1: Prompt 工程** - 先在測試環境調校 Groq Prompt，確保能穩定輸出純 JSON 結構。
- **Phase 2: 核心後端邏輯** - 實作 YouTube API 搜尋功能，並同時建立 Firebase 快取檢查與寫入機制。
- **Phase 3: 模組整合** - 封裝「心情輸入 -> Groq 分析 -> 快取/搜尋 -> 寫入資料庫」的完整工作流程。
- **Phase 4: 前端與 UI 實作** - 依照日系可愛、溫馨風格實作前端介面，並整合 YouTube IFrame API 完成最終影音串接。
